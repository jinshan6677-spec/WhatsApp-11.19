# 安全的翻译架构迁移计划

## ⚠️ 重要发现

经过分析，我发现：

### 新框架的依赖关系

```
ViewTranslationIntegration (新框架)
  ↓ 调用
TranslationIntegration.injectScripts() (旧框架) ⚠️
  ↓ 注入
contentScript.js (旧框架) ⚠️
```

**结论**：新框架**没有自己的翻译内容脚本**，仍然依赖旧的 `contentScript.js`！

## 问题分析

如果删除旧的翻译文件，会导致：
1. ❌ `contentScript.js` 被删除 → 翻译功能完全失效
2. ❌ `TranslationIntegration.js` 被删除 → 无法注入脚本
3. ❌ `preload-view.js` 被删除 → 无法暴露 API

## 两个选择

### 选项 A：安全迁移（推荐）⭐

**只删除真正重复的文件**，保留新框架依赖的文件。

**可以安全删除**：
- ✅ `src/translation/ipcHandlers.js` - 已删除（新框架已替代）

**必须保留**：
- ⚠️ `src/translation/contentScript.js` - 新框架仍在使用
- ⚠️ `src/translation/translationService.js` - 新框架仍在使用
- ⚠️ `src/translation/engines/` - 新框架仍在使用
- ⚠️ `src/managers/TranslationIntegration.js` - 新框架仍在使用
- ⚠️ `src/preload-view.js` - 新框架仍在使用

**结果**：
- ✅ 翻译功能继续工作
- ✅ 使用新框架的架构
- ✅ 保留稳定的业务逻辑

### 选项 B：完全重写（高风险）⚠️

**需要做的事情**：

1. **在新框架中重写翻译内容脚本**：
   - 创建 `src/presentation/translation/contentScript.js`
   - 重写 4000+ 行的翻译逻辑
   - 重新实现所有功能

2. **在新框架中重写注入机制**：
   - 在 `ViewTranslationIntegration` 中实现脚本注入
   - 不再依赖旧的 `TranslationIntegration`

3. **在新框架中重写 Preload 脚本**：
   - 创建新的 Preload 脚本
   - 暴露翻译 API

4. **大量测试**：
   - 测试所有翻译功能
   - 测试所有配置选项
   - 测试所有边缘情况

**时间估计**：2-4 周

**风险**：⚠️ 极高（翻译功能可能完全失效）
